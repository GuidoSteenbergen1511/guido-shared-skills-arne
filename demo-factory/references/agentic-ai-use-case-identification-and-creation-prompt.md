# Analysis Framework

## Interview Analysis Process

Analyze the provided interview transcripts to extract:

- **Specific pain points** mentioned by stakeholders
- **Current manual processes** that consume time or resources
- **Document types and workflows** referenced in conversations
- **Repetitive tasks** that occur frequently
- **Time estimates** for current processes (e.g., "takes me 20 minutes per report")
- **Quality issues** due to manual handling (errors, inconsistencies)
- **Desired automation outcomes** expressed by interviewees
- **Industry-specific terminology** and specialized workflows
- **Stakeholder roles** and their unique challenges
- **Integration touchpoints** with existing systems

## Document Repository Assessment

After identifying pain points from interviews, systematically examine the document repository to locate:

- **Input documents** (what teams receive/process)
- **Output documents** (what teams create/deliver)
- **Template documents** (standardized formats used)
- **Historical examples** (completed workflows from past projects)
- **Variation patterns** (different formats for similar processes)
- **Process documentation** (SOPs, guidelines, templates)
- **Communication patterns** (emails, meeting notes, reports)
- **Data sources** (spreadsheets, databases, external feeds)
- **Regulatory requirements** (compliance documents, audit trails)

### Style Mimicry & Template Extraction Requirements

When assessing documents, explicitly extract the organization's style rules and encode them in output templates:

- Section hierarchy: exact headers, subheaders, numbering, and ordering
- Bullet glyphs and nesting: capture symbols (e.g., •, o, -) and indentation conventions
- Tables and matrices: column names, totals/∑ rows, percentage rows, growth columns, alignment rules, and number formatting (parentheses for negatives, X.X% for percentages)
- Visual patterns: boxes, info panes, diagrams/frames referenced in text (describe how to reproduce them in markdown or narrative)
- Tone and register: formal/informal, Dutch/English, phrasing patterns used in originals
- Legal/standard clauses: reusable boilerplate sections and where to include them

Deliver concrete "Output Template" sections in each CustomGPT that mirror these patterns, and include 1-2 short, anonymized examples sourced from the repository (or synthetic examples that strictly follow the style).

**CRITICAL**: Do NOT attempt to write full prompts directly into the master file. Large prompts (5,000-10,000 characters each) will create token-intensive operations and potential errors. Instead, use this proven approach:

**Why this approach**:

- Creates clear injection boundaries with regex-searchable markers
- Allows file structure to be reviewed before content injection
- Enables systematic, automated prompt injection via command-line tools
- Prevents token overflow and ensures consistent formatting

#### Step 2: Inject Full Prompts via Command-Line Automation

Use PowerShell (Windows) or bash (Mac/Linux) to inject actual prompt content:

**PowerShell Injection Command Pattern** (Windows):

```powershell
cd "[Project Path]"
$masterFile = "demos\Knowledge_Management\MASTER_All_GPT_Configurations.md"
$content = Get-Content -Path $masterFile -Raw
$demoContent = Get-Content -Path "demos\Demo1_[UseCase]\ChatGPT_Instructions.md" -Raw
$pattern = '(?s)(<!-- START_PROMPT_DEMO1 -->).*?(<!-- END_PROMPT_DEMO1 -->)'
$replacement = "`$1`n`n$demoContent`n`n`$2"
$content = $content -replace $pattern, $replacement
Set-Content -Path $masterFile -Value $content -NoNewline
Write-Host "Demo 1 injected successfully!"
```

**Bash Injection Command Pattern** (Mac/Linux):

```bash
cd "[Project Path]"
MASTER_FILE="demos/Knowledge_Management/MASTER_All_GPT_Configurations.md"
DEMO_FILE="demos/Demo1_[UseCase]/ChatGPT_Instructions.md"
DEMO_CONTENT=$(cat "$DEMO_FILE")
perl -i -pe "BEGIN{undef $/;} s|(<!-- START_PROMPT_DEMO1 -->).*?(<!-- END_PROMPT_DEMO1 -->)|\$1\n\n$DEMO_CONTENT\n\n\$2|smg" "$MASTER_FILE"

```
